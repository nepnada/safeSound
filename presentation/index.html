Bonjour. Nous vous presentons SafeSound, un bracelet IoT intelligent destine aux personnes sourdes et malentendantes. Le bracelet ecoute l'environnement en continu, classifie les sons critiques en temps reel directement sur le microcontroleur, et alerte l'utilisateur via des LEDs colorees et des vibrations. Aucune connexion cloud n'est necessaire : tout le traitement se fait sur l'appareil, avec un modele de seulement 36.4 kilo-octets qui atteint 95.4% de precision.

Knowledge Distillation (Hinton et al., 2015)

La Knowledge Distillation est une technique introduite par Geoffrey Hinton, Oriol Vinyals et Jeff Dean dans l'article "Distilling the Knowledge in a Neural Network" (2015). Le principe : un grand modele "Teacher" transfere ses connaissances a un petit modele "Student" via des distributions de probabilites douces (soft labels).

A temperature T=1, le Teacher dit "c'est une alarme incendie a 95%". A T=6, il dit "c'est 60% alarme, 25% klaxon, 15% fond" — cette information de similarite inter-classes est invisible dans les labels classiques one-hot, mais extremement utile pour un petit modele.

// Fonction de perte de distillation loss = 0.7 * KL(teacher_soft, student_soft) * T2 + 0.3 * CrossEntropy(labels, student) // T=6 adoucit la distribution du Teacher // alpha=0.7 : 70% du signal vient du Teacher
DS-CNN (Zhang et al., 2017 — "Hello Edge")

L'architecture DS-CNN (Depthwise Separable CNN) est issue de l'article "Hello Edge: Keyword Spotting on Microcontrollers". Elle utilise des convolutions separables en profondeur : au lieu d'un filtre 3x3 applique sur tous les canaux simultanement, on applique d'abord un filtre 3x3 par canal (depthwise), puis une convolution 1x1 (pointwise) pour combiner les canaux.

Cette factorisation reduit le nombre d'operations de ~8x pour un noyau 3x3 avec 64 canaux, ce qui est essentiel pour une inference en <2ms.

// Architecture du Student DS-CNN Conv2D(32) → BN → ReLU → MaxPool DS Block 1: DW(3x3) → BN → PW(64) → MaxPool DS Block 2: DW(3x3) → BN → PW(64) → MaxPool DS Block 3: DW(3x3) → BN → PW(128) GlobalAveragePooling → Dropout(0.3) → Dense(6) // 18,086 parametres total
Notre pipeline ML utilise deux techniques avancees issues de la recherche. La Knowledge Distillation, introduite par Hinton, Vinyals et Dean en 2015, permet de compresser un grand modele Teacher de 423 mille parametres en un petit Student de seulement 18 mille parametres — un ratio de compression de 23x. Le Teacher apprend d'abord les relations entre classes, puis transfère cette connaissance via des probabilites adoucies a temperature T=6. L'architecture du Student est un DS-CNN, issu de l'article "Hello Edge" de Zhang et al. en 2017, specialement concu pour l'inference sur microcontroleurs. Les convolutions separables en profondeur reduisent les operations par un facteur 8. Enfin, la quantification QAT (Quantization-Aware Training) convertit tous les poids et activations en INT8, divisant la taille par 4 tout en maintenant, voire ameliorant, la precision grace a la regularisation supplementaire.
05 Signal Processing

Traitement du signal : De l'audio brut au tenseur INT8

1. Capture audio

Le micro INMP441 echantillonne a 16 kHz en mono via I2S. Chaque fenetre d'analyse fait 1 seconde (16,000 echantillons) avec un chevauchement de 50% (stride 500ms).

2. Spectrogramme Log-Mel

STFT avec fenetre FFT de 512 echantillons (32ms) et hop de 256 (16ms). Les coefficients de puissance sont projetes sur 64 filtres Mel entre 60 Hz et 7600 Hz, puis convertis en echelle logarithmique (dB). Pourquoi Log-Mel et non MFCC ? Les MFCC appliquent une DCT qui detruit la localite spatiale — mauvais pour un CNN.

3. Normalisation [0, 1]

Normalisation min-max par fenetre. Sortie : tenseur float32 de forme (1, 64, 63, 1) — 64 canaux Mel, 63 frames temporels.

4. Quantification INT8

Transformation affine : x_int8 = x_float / scale + zero_point. Le tenseur float32 de 15.8 KB devient un tenseur INT8 de 4.0 KB — reduction de 4x avec perte minimale.

# Feature extraction (Python, identique au firmware) import librosa # Parametres (identiques dans firmware/esp32/main.ino) SAMPLE_RATE = 16000 N_FFT = 512 # fenetre FFT = 32ms HOP_LENGTH = 256 # hop = 16ms N_MELS = 64 # canaux Mel F_MIN, F_MAX = 60, 7600 # 1. Spectrogramme Mel (puissance) mel = librosa.feature.melspectrogram( y=audio, sr=SAMPLE_RATE, n_fft=N_FFT, hop_length=HOP_LENGTH, n_mels=N_MELS, fmin=F_MIN, fmax=F_MAX ) # 2. Conversion en echelle logarithmique log_mel = librosa.power_to_db(mel, ref=np.max) # 3. Normalisation min-max normalized = (log_mel - log_mel.min()) \ / (log_mel.max() - log_mel.min()) # 4. Quantification INT8 x_int8 = (normalized / scale + zero_point).astype(np.int8) # Forme finale : (1, 64, 63, 1)

Choix IoT et justification
BLE 5.0 (pas Wi-Fi) : Consommation 10x inferieure au Wi-Fi. Le bracelet n'a pas besoin d'IP propre — le smartphone sert de bridge.
MQTT (pas HTTP) : Protocole leger pub/sub, ideal pour IoT. Overhead minimal par message (~2 bytes header). QoS configurable.
Node-RED : Programmation visuelle des flux. Permet de configurer les regles d'alerte et les notifications sans code.
Architecture edge-first : L'inference est locale. Le cloud est optionnel — uniquement pour la notification de l'aidant. Le bracelet fonctionne meme sans telephone.
Cote IoT, l'architecture suit un modele edge-first. L'inference est entierement locale sur l'ESP32. Le cloud n'intervient que pour notifier un aidant distant. Le bracelet communique avec le smartphone via BLE 5.0, choisi pour sa consommation 10 fois inferieure au Wi-Fi. Le smartphone sert de bridge BLE vers MQTT. Mosquitto est le broker MQTT, et Node-RED orchestre les flux. Les topics MQTT suivent une convention hierarchique : classification, alerte, status device, notification, et configuration des seuils a distance. Le point cle : le bracelet continue de fonctionner meme sans telephone. Les alertes LED et vibration sont toujours actives.
